\section{Configuration And Operations}
\label{sec:ConfigurationOperations}

\subsection{Scaffolder invocation}

The repository includes a convenience script at \filepath{durga}:

\begin{verbatim}
./durga path/to/process.bpmn [flags...]
\end{verbatim}

It auto-builds the JAR if missing and passes all arguments through to the
scaffolder. Equivalent to:

\begin{verbatim}
mvn -q package -DskipTests
java -jar target/durga-1.0-SNAPSHOT.jar path/to/process.bpmn [flags...]
\end{verbatim}

Available flags: \texttt{--dry-run}, \texttt{--out <dir>},
\texttt{--transactions}, \texttt{--separate-workers}, \texttt{--strimzi},
\texttt{--connect}, \texttt{--event-topic <topic>}, \texttt{--process-id <id>},
\texttt{--package <pkg>}, \texttt{--retention <hours>}.

\texttt{--retention <hours>} sets Kafka topic retention in hours
(defaults to 168\,h/7\,days with a warning). Use \texttt{-1} for
infinite retention. The canonical event topic (configured via
\texttt{--event-topic}) always has infinite retention; the
\texttt{\%process\%\_state} topic always uses compaction. Wired
into both \texttt{topics.sh} and Strimzi \texttt{topics.yml}.
Accepts \texttt{h}, \texttt{d}, \texttt{w}, \texttt{m} suffixes
for hours, days, weeks, or 30-day months.

\texttt{--package <pkg>} sets the Java package for generated code
(defaults to \texttt{org.gautelis.durga.generated} with a warning).
Operational classes are placed directly in the package; probe and
exerciser classes in \texttt{<pkg>.probes}.

\texttt{--event-topic <topic>} sets the physical Kafka topic name for
canonical lifecycle events (defaults to \texttt{process-events}). Use
per-pipeline names such as \texttt{process-events-orders} to isolate
pipeline traffic and control retention independently. The logical channel
names \texttt{process-events} and \texttt{process-events-monitor} in the
generated code stay unchanged; only the physical topic mapping in
\texttt{application.yml} is affected. The monitoring topology accepts
the topic as a command-line argument.

\texttt{--process-id <id>} overrides the process identifier read from the
BPMN file. This is the name used in topic names, consumer group IDs,
generated class names, and the \texttt{durga-<id>} consumer group prefix
for the \texttt{ProcessStateStore}. Pinning the process ID keeps Kafka
group coordination stable if the BPMN tool regenerates with different
element identifiers.

\subsection{Local runtime baseline}

The normal local environment consists of:
\begin{itemize}
\item the bundled Kafka setup under \filepath{setup/},
\item generated topic configuration in the scaffolded project,
\item the optional monitoring app and dashboard.
\end{itemize}

The repository is intentionally optimized for a local single-broker loop so the generated
processes and monitoring projections can be explored quickly.

\subsection{Important configuration surfaces}

The main configuration locations are:
\begin{itemize}
\item repository-level runtime defaults in \filepath{src/main/resources/application.yml},
\item generated-project channel configuration in the scaffolded \texttt{application.yml},
\item local Kafka infrastructure in \filepath{setup/docker-compose.yml},
\item generated topic setup scripts in scaffolded output directories.
\end{itemize}

\subsection{Operational expectations}

In normal local startup:
\begin{itemize}
\item Kafka UI may log temporary connection failures before the broker is ready,
\item generated runtimes rely on explicit topics and channels rather than implicit discovery,
\item the monitoring app may report a transitional Streams state before its stores become queryable.
\end{itemize}

\subsection{Where state lives}

Durga uses several layers of state:
\begin{itemize}
\item transport history on execution topics,
\item canonical lifecycle history on \texttt{process-events},
\item materialized latest state and counts in Kafka Streams stores and corresponding topics,
\item local waiting and cancellation bookkeeping inside some generated handlers.
\end{itemize}

\subsection{Recommended local operator loop}

For practical local work:
\begin{enumerate}
\item start Kafka,
\item build the repository or generated project,
\item create topics,
\item run the generated process or a demo publisher,
\item start the monitoring app if query visibility is needed,
\item inspect lifecycle events, counts and latency.
\end{enumerate}
