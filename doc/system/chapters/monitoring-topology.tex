\section{Monitoring Topology}
\label{sec:MonitoringTopology}

\subsection{Purpose}

The monitoring subsystem is a separate runtime concern from the generated process
execution graph. Generated handlers emit canonical lifecycle events; the monitoring
topology consumes those events and turns them into queryable operational views.

Execution topics show how work is routed. The monitoring topology answers:
\begin{itemize}
\item where is a specific process instance now,
\item how many instances are in a given state,
\item which activities are taking the longest,
\item which instances appear stuck,
\item what are the coarse trend counts for starts, completions, and failures.
\end{itemize}

\subsection{Topology structure}

The entry point is \texttt{ProcessMonitoringTopology.buildTopology()}. It consumes \\
\texttt{process-events}, filters null keys and values, and forks the stream into
five parallel sub-topologies, each with a dedicated local state store:

\begin{description}
\item[Instance state] Aggregates events into \texttt{ProcessStateView} per instance.
Materialized in \\ \texttt{process-state-store}, output to \texttt{process-state}.
A filtered subset (active instances) is output to \texttt{process-active-state}.
\item[State counts] Regroups state views by \texttt{processId:state} key and counts
instances. Materialized in \texttt{process-state-counts-store}, output to
\texttt{process-state-counts}.
\item[Activity latency] Feeds events through a custom \texttt{ActivityLatencyProcessor}
that records \\ \texttt{ACTIVITY\_ENTERED} timestamps in a persistent key-value store
(\texttt{process-activity-entry-store}) and, on later completion or termination events,
emits duration deltas. These deltas are aggregated into rolling \texttt{ActivityLatencyStats}
per activity, then mapped to \texttt{ActivityLatencySummary} and output to
\texttt{process-latency}.
\item[Trends] Filters to \texttt{PROCESS\_STARTED}, \texttt{PROCESS\_COMPLETED},
and \texttt{PROCESS\_FAILED} events, maps them to hourly trend bucket keys,
counts per bucket, and outputs \texttt{ProcessTrendPoint} records to
\texttt{process-trends}. Materialized in \texttt{process-trends-store}.
\item[Queryable replicas] Five \texttt{builder.globalTable()} calls consume the
output topics above, each materialized as a global table with a \texttt{-global-store}
suffix. These are full replicas on every monitoring instance and are what the query
service reads.
\end{description}

The five local stores drive computation — aggregation, counting, and temporal joins.
The five global stores serve queries. The output topics (\texttt{process-state},
\texttt{process-state-counts}, \\ \texttt{process-active-state}, \texttt{process-latency},
\texttt{process-trends}) decouple computation from query access: any monitoring instance
can spin up and receive a full read replica without recomputing.

\subsection{Latency computation}

Activity latency uses a temporal join pattern without windowed stream-stream joins.
The \\ \texttt{ActivityLatencyProcessor} stores each \texttt{ACTIVITY\_ENTERED} event's
timestamp under the key \texttt{instanceId:activityId} in the persistent
\texttt{process-activity-entry-store}. When a later \texttt{ACTIVITY\_COMPLETED},
\texttt{ACTIVITY\_ESCALATED}, \texttt{ACTIVITY\_CANCELLED}, \texttt{GATEWAY\_TAKEN}, \\ 
\texttt{PROCESS\_COMPLETED}, or \texttt{PROCESS\_FAILED} event arrives for the same
key, the processor computes the duration and emits an \texttt{ActivityLatencyDelta}.

These deltas are grouped by a canonical activity key (processId, version, activityId,
event type) and aggregated into \texttt{ActivityLatencyStats} — a running count,
sum, min, and max — then mapped to \texttt{ActivityLatencySummary} for output.

\subsection{Processing guarantees}

The topology is configured with \texttt{AT\_LEAST\_ONCE} semantics. This trades
exactly-once guarantees for higher throughput and lower latency. State views may
transiently reflect duplicate or stale updates, which is acceptable for a monitoring
read model. The state directory is configured to \filepath{target/kafka-streams-state}
— suitable for local development, but should be made configurable for production use.

\subsection{Query access pattern}

The query service (\texttt{ProcessMonitoringQueryService}) accesses only the global
table stores via \texttt{KafkaStreams.store()}. It uses point lookups for instance
state and full-store scans for counts, latency, trends, and stuck detection — filtering
in memory by process ID where needed. No interactive queries or cross-instance RPC is
required, since global tables replicate the entire content to every monitoring instance.

\subsection{Relationship to demos}

Repository-level demo publishers and scenario runners emit synthetic lifecycle
events directly to \\ \texttt{process-events}, making the monitoring subsystem
independently testable without a full generated process runtime.

\subsection{Current shape}

The monitoring path is now stronger than the original local-only read model:
\begin{itemize}
\item queryable state is replicated from monitoring topics into global read-side stores, so each instance carries a full local copy of the monitoring model,
\item latency summaries are maintained incrementally via a custom processor rather than derived from scanning latest-instance state,
\item active-instance queries use a dedicated active-state index instead of the full state projection,
\item coarse history is maintained as bucketed trend counts for process starts, completions, and failures.
\end{itemize}

History is intentionally coarse — Durga supports basic trend reporting but is
optimized more for current-state inspection than long-retention analytical
reporting.
