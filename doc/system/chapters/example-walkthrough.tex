\section{Example Walkthrough}
\label{sec:GeneratedProjectsExample}

\subsection{Model}

The reference model is \filepath{src/test/resources/bpmn/invoice_receipt.bpmn}:
start, service work, review, approve/reject, notify.

\subsection{Generation}

\begin{verbatim}
./durga src/test/resources/bpmn/invoice_receipt.bpmn
\end{verbatim}

Or directly:

\begin{verbatim}
mvn -q clean package
java -jar target/durga-1.0-SNAPSHOT.jar \
  src/test/resources/bpmn/invoice_receipt.bpmn
\end{verbatim}

Output lands in \texttt{generated/} with Java sources, \texttt{pom.xml},
Kafka topic declarations, helper scripts, and a generated \texttt{README.md}.

\subsection{Generated contents}

\begin{itemize}
\item worker services for automatic task execution,
\item waiting-state handlers for human-completion steps,
\item a starter class that publishes the first process event,
\item a process orchestrator that detects terminal completions,
\item helper publishers and watchers for demos and testing,
\item Kafka bindings in \texttt{application.yml}.
\end{itemize}

\subsection{Runtime flow}

\begin{enumerate}
\item the starter publishes the initial process event,
\item task workers consume input topics,
\item each step emits lifecycle events to \texttt{process-events},
\item routing services move the process to the next task or end event,
\item the orchestrator emits \texttt{PROCESS\_COMPLETED}.
\end{enumerate}

Richer models add boundaries, subprocess scopes, and event subprocesses but
follow the same pattern: handlers move through Kafka topics while
\texttt{process-events} carries the lifecycle truth.

\subsection{Exploring the generated process}

\begin{itemize}
\item \texttt{demo-scenario.sh} exercises a whole process path,
\item \texttt{send-task-input.sh} pushes a specific task input,
\item \texttt{complete-task.sh}, \texttt{fail-task.sh}, \texttt{escalate-task.sh}
  force specific task outcomes,
\item \texttt{watch-process-events.sh}, \texttt{watch-task-output.sh}
  expose the runtime flow.
\end{itemize}
