\section{Monitoring Interfaces}
\label{sec:MonitoringInterfaces}

The monitoring subsystem exposes these interfaces over the same Kafka Streams state:
\begin{itemize}
\item an HTTP API and embedded SPA,
\item a command-line client,
\item a browser HTML dashboard.
\end{itemize}

\subsection{HTTP service}

Entry point: \texttt{MonitoringContainer}. Query layer:
\texttt{ProcessMonitoringQueryService} and \texttt{ProcessMonitoringHttpServer}.

Endpoints:
\begin{itemize}
\item \texttt{/health} --- Kafka Streams readiness,
\item \texttt{/processes/list} --- all known process IDs (from counts + model registry),
\item \texttt{/process} --- the single process ID when running in single-process mode,
\item \texttt{/instances/\{processInstanceId\}} --- latest instance view,
\item \texttt{/processes/\{processId\}/counts} --- process-specific state counts,
\item \texttt{/processes/\{processId\}/latency} --- activity latency summaries,
\item \texttt{/processes/\{processId\}/trends} --- bucketed start/completion/failure history,
\item \texttt{/counts} --- global state-count view,
\item \texttt{/stuck} --- stuck-instance detection,
\item \texttt{/diagram?processId=\textit{id}} --- BPMN 2.0 XML from the model cache,
\item \texttt{/metrics} --- Prometheus-format metrics,
\item \texttt{/dashboard} --- browser HTML dashboard.
\end{itemize}

All paths accept an \texttt{/api/} prefix. The diagram endpoint queries the
\texttt{process-models-store} first, then falls back to a local BPMN directory
(\texttt{\{bpmnDir\}/\{processId\}.bpmn}), and finally to a single BPMN file
for development convenience.

The HTTP API is designed for internal operational use. Put it behind the same
network and identity controls used for other production admin interfaces before
exposing it outside a trusted environment.

\subsection{CLI client}

The repository includes a convenience script at \filepath{durga-monitor}:

\begin{verbatim}
./durga-monitor health
./durga-monitor counts invoice_receipt
./durga-monitor latency invoice_receipt
./durga-monitor stuck invoice_receipt 300
./durga-monitor instance <processInstanceId>
\end{verbatim}

Defaults to \texttt{http://localhost:8081}. Override with
\texttt{DURGA\_URL}. Equivalent to:

\begin{verbatim}
java -cp target/durga-0.1.0-beta.1.jar \
  org.gautelis.durga.monitoring.ProcessMonitoringClient \
  http://localhost:8081 <command>
\end{verbatim}

\subsection{Dashboard}

The Svelte SPA is built from \filepath{monitoring-ui/} and served directly
from the monitoring backend. It shows:
\begin{itemize}
\item a process inventory with all known process IDs and aggregate KPIs,
\item per-process drill-down: state mix, lifecycle trends, activity latency, stuck instances,
\item instance detail lookup,
\item a BPMN diagram viewer with color-coded activity overlays (latency and state).
\end{itemize}

The diagram viewer uses \texttt{bpmn-js} with built-in zoom (scroll wheel),
pan (grab tool), and fit-to-viewport controls.

The HTML dashboard (\texttt{/dashboard}) is a standalone server-rendered page
that polls the same HTTP API. It serves as a lightweight alternative when the
SPA is unavailable.

\subsection{How the interfaces fit together}

All interfaces consume the same materialized state:
\begin{enumerate}
\item processes self-register by publishing BPMN models to \texttt{process-models},
\item lifecycle events flow through \texttt{process-events-*},
\item Kafka Streams materializes state, counts, latency, trends, and the model cache,
\item the HTTP layer exposes that materialized state,
\item the CLI and dashboard consume the same HTTP API.
\end{enumerate}

The operator workflow is:
\begin{itemize}
\item deploy a process --- it auto-registers its model on boot,
\item the dashboard shows the new process in the inventory,
\item publish a scenario or task outcome,
\item watch lifecycle events,
\item query one instance or inspect counts and latency,
\item view the BPMN diagram with live overlays.
\end{itemize}
