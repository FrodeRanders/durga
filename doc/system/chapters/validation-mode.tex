\section{Validation Mode}
\label{sec:ValidationMode}

\subsection{Purpose}

Validation mode runs a not-yet-released implementation of a process against real
input, alongside the current (or prior) version, and reports where the new version
produces the same output and where it diverges. It answers a concrete release
question: \emph{if I deploy this candidate, what changes downstream?} Sometimes an
identical input$\rightarrow$output is expected (a refactor); sometimes divergence is
intended (a functional correction). Validation mode does not judge --- it produces the
evidence and lets a human decide.

\subsection{A complete shadow, not extra workers}

\texttt{--validation} is a \emph{mode}, not an add-on. Scaffolding a model with
\texttt{--validation} generates the \emph{whole} process again, wired differently ---
not an additional shadow worker beside each production worker. The shadow process:
\begin{itemize}
\item \textbf{reads the same live production input} that the running process consumes,
  through a \emph{dedicated consumer group} (\texttt{\%processId\%-validation} for the
  Java target; \texttt{\%processId\%-\%taskId\%-validation} per worker for the Rust
  target), so the production consumer offsets --- the current input index --- are never
  advanced or disturbed;
\item starts at \texttt{latest} by default (live-concurrent shadowing) and can be
  pointed at recent history via \texttt{DURGA\_VALIDATION\_OFFSET\_RESET} (a bounded
  replay near the tail --- not from the very beginning, since old input may not be
  representative);
\item \textbf{writes nothing to a real process topic.} Each task's output is diverted to
  a per-task \texttt{\%processId\%\_\%taskId\%\_output-validation} topic and its DLQ to
  \texttt{\%processId\%\_\%taskId\%\_dlq-validation}; lifecycle events go to the per-process
  \texttt{process-events-\%processId\%-validation} topic instead of the live \texttt{process-events-\%processId\%};
\item runs each plugin with side effects suppressed (Section~\ref{sec:ValidationMode}
  below).
\end{itemize}

\subsection{Per-task isolation}

Validation is handled \emph{per task}. Each task in the shadow is fed the production
input \emph{for its own position} in the chain --- the production predecessor's output
topic --- not the shadow's own reworked output. Candidate outputs are therefore never
chained together: a divergence introduced by a modified early task cannot cascade into
and contaminate the comparison of later tasks. Every task is compared in isolation
against production for the identical shared input. (The shadow's own inter-task
\texttt{-validation} outputs are consequently unread --- they exist only so the shadow
never writes to a real topic.)

\subsection{Side-effect suppression}

The generated worker calls the plugin with a
\texttt{PluginExecutionContext}: \texttt{production()} in a normal build,
\texttt{validation()} in a validation build. The single structured entry point is
\texttt{PluginResult executeWithResult(byte[] payload, String config,
PluginExecutionContext context)} (mirrored by \texttt{execute\_with\_result} in the Rust
trait). Pure transforms ignore the context; plugins that perform substantial side
effects override it to suppress them when \texttt{context.validationMode()} is true while
still returning the response they \emph{would} have produced. For example
\texttt{ObjectStoreCollector} computes and returns the handle it would have written but
writes no object, marking the URI \texttt{validation:not-stored/\ldots}. For handle-aware
(\texttt{materialize}) execution, \texttt{PluginExecutionSupport.executeSandboxed} runs
the transform but writes no object to the store, returning a synthetic handle whose
content hash still reflects the real output so volatile fields (uri, timestamps) can be
excluded by comparison ignore paths.

The monitor's \texttt{ValidationTopology} pairs each shadow task's
\texttt{ACTIVITY\_COMPLETED} event (from the per-process \texttt{process-events-\%processId\%-validation} topic)
against the production \texttt{ACTIVITY\_COMPLETED} for the same input (from the
per-process \texttt{process-events-*} streams, excluding validation), keyed identically
on \texttt{processId:activityId:processInstanceId}. Both sides carry the same instance id
(the shadow replays the same start records), so they join naturally; the merge is a
single aggregate, robust to arrival order. A candidate seen before any prior output
yields \texttt{PRIOR\_MISSING}, superseded when the prior output later arrives.

The candidate side subscribes to \texttt{process-events-*-validation}, while the
production side subscribes to \texttt{process-events-*} excluding \texttt{*-validation}.
This keeps shadow lifecycle events out of production state and alarm projections while
allowing the comparator to pair each shadow task with its production counterpart.

Comparison is a structural, order-sensitive JSON diff (\texttt{JsonComparison}) with
configurable \emph{ignore paths} (dot-notation, with \texttt{*} and \texttt{[*]}
wildcards; set via \texttt{durga.validation.ignore.paths}) so volatile fields such as
generated timestamps or identifiers do not register as divergence. Each comparison
produces a \texttt{ValidationResult} with one of four outcomes:

\begin{description}
\item[\texttt{EQUAL}] candidate output identical to prior after ignore paths.
\item[\texttt{DIFF}] candidate output differs; the located differences (path, kind,
  prior value, candidate value) are attached.
\item[\texttt{PRIOR\_MISSING}] no production output found for the shared input.
\item[\texttt{CANDIDATE\_ERROR}] the candidate failed or requested a non-success
  error strategy.
\end{description}

Results are written to \texttt{validation-results} and materialized in the
\texttt{validation-results-store} global table for query.

\subsection{Topics and naming}

\begin{itemize}
\item dedicated consumer group --- \texttt{\%processId\%-validation} (Java) or
  \texttt{\%processId\%-\%taskId\%-validation} (Rust) --- reading the live production
  chaining topics;
\item \texttt{\%processId\%\_\%taskId\%\_output-validation} and
  \texttt{\%processId\%\_\%taskId\%\_dlq-validation} --- per-task shadow output and DLQ;
\item \texttt{process-events-\%processId\%-validation} --- the per-process topic carrying the shadow's
  lifecycle events, consumed by the comparator;
\item \texttt{validation-results} --- comparison results, materialized in
  \texttt{validation-results-store}.
\end{itemize}

The main state and fault/alarm topologies subscribe to
\texttt{process-events-(?!.*-validation).*}, so they never ingest shadow events: a
running validation cannot trip phantom stuck/cascade alarms or pollute production state.

\subsection{Reporting interfaces}

The monitor exposes the report under \texttt{/api/validation/}:
\begin{itemize}
\item \texttt{/api/validation/summary?processId=\textit{id}} --- per-task counts
  (equal / diff / prior-missing / candidate-error / total); \texttt{processId} is
  optional (omit it for all processes),
\item \texttt{/api/validation/results?processId=\&taskId=\&status=} --- individual
  comparisons, optionally filtered,
\item \texttt{/api/validation/instances/\{instanceId\}} --- all validated
  tasks for one instance.
\end{itemize}

The dashboard adds a \emph{Validation Report} panel to the per-process drill-down: a
per-task summary table and, per instance, an expandable comparison showing the shared
input, prior output, candidate output, and a highlighted diff table. \texttt{EQUAL}
comparisons (the common, uninformative case) are hidden by default and opt-in via a
\emph{show equal} toggle; their count is always shown. Prometheus metrics
\texttt{durga\_validation\_total}, \texttt{durga\_validation\_diff}, and
\texttt{durga\_validation\_candidate\_error} are exported per
\texttt{process\_id,task\_id} on \\ \texttt{/api/metrics}.

\subsection{Java and Rust parity}

Both code-generation targets emit the same \texttt{ProcessEvent} lifecycle records to the
same \texttt{process-events-\%processId\%-validation} topic, and both chain tasks through \emph{identical}
topics (Section~\ref{sec:RustTarget}). A shadow is therefore
wire-interchangeable with production regardless of target: a Rust-generated shadow can
validate a running Java process (and vice versa), task by task --- a convenient way to
verify the two targets against each other. There is no separate candidate wire record and
no separate validation binary: validation is the whole process, generated once and wired
to the validation topics.
